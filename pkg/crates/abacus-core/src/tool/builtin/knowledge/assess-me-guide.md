# Assess Me — 认知状态审计详细指南

## 6 问审计详解

### 1. Goal（目标）
- 任务是什么？"完成"长什么样？
- **常见错误**：把"在做 X"当目标（这是手段不是目标）

### 2. Progress（进展）
- 已完成什么？还差什么？
- **常见错误**：复述努力而非结果（"我试了 A、B、C" ≠ "A 确认了 X，B 排除了 Y"）

### 3. Gaps（盲点）
- 缺什么信息？哪些数据没拿到？
- **检查**：有没有你"觉得不需要看"但其实没看过的文件/日志？

### 4. Assumptions（假设）
- 哪些未验证的信念在驱动你的 approach？
- **不能为空！** 总有假设。常见隐含假设：
  - "这个文件是最新的"（可能被改过）
  - "这个函数按文档工作"（可能有 bug）
  - "环境变量是正确的"（可能配错）
  - "依赖版本没变"（可能升级了）

### 5. Blocker（阻塞）
- 具体阻塞是什么？
- **模糊阻塞** = 还没找到真正的阻塞
- 好的阻塞："cargo test 在 line 42 panic，因为 xxx 变量为 None"
- 坏的阻塞："测试不通过"

### 6. Recovery（恢复路径）
- 如果卡住了，应该做什么不同的？
- **必须是具体行动**，不是"继续试"
- 好的恢复："用 debug! 打印 xxx 变量，确认它在哪个分支被设为 None"
- 坏的恢复："再仔细看看代码"

## 批判性审视清单

读回 temp file 后检查：
- [ ] Progress 是否只复述努力而非结果？
- [ ] Blocker 是否具体而非模糊？
- [ ] Recovery 是否是具体行动而非"继续试"？
- [ ] Assumptions 是否为空？（不应为空）
- [ ] 是否有未检查的盲点？

## 触发条件补充

除了用户质疑，以下场景也应触发：
- 连续 3 次修复尝试无进展
- 工具返回的结果与预期完全不符
- 同时存在 3+ 个可能的假设
- 代码改动后不确定影响范围
